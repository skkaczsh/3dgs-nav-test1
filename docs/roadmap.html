🧠 知识层

坐标系变换树

MANIFOLD 设备 5 个坐标系之间的变换关系：

  World
    │
    │  body_T_lidar（逐帧，calib_online.yaml）
    │
    ▼
  Body ←── Til（IMU 与 Body 对齐，calib_online_final.yaml）
    │
    │  inv(Til) = T_body_lidar
    │
    ▼
  LiDAR ←── .lx 点云在此坐标系
    │
    │  inv(Tcl_0/1/2) = T_lidar_cam（在线标定，每帧微调）
    │
    ▼
  Cam{0,1,2}

相机世界位姿计算：T_world_cam = body_T_lidar @ Til @ inv(Tcl_*) @ OPENCV_TO_OPENGL

在线标定 固定外参 img_pos 风险

img_pos.txt 列含义

7,631 行 × 19 列（空格分隔，逗号为千位分隔符）：

列1帧号 (0~7630)
列2Unix 时间戳
列3-5IMU 位置 (x,y,z) — 非世界坐标！
列6-9四元数 (qw,qx,qy,qz)
列10四元数归一化因子
列11-13角速度 (ωx,ωy,ωz)
列14,16,18分隔常量 (10)
列15,17,19帧分组号

⚠️ 列3-5 是 IMU 局部坐标增量，绝对不能当世界坐标用于 transforms.json 构建！

img_pos 风险 在线标定（世界坐标权威来源）

.lx Section 格式回顾

每个 PLY section 对应 .lx 的一帧数据。Section 结构：

Header48 bytes（版本/标定数据/帧号/参数）
Point Count4 bytes (uint32 LE)
PointsN × 16 bytes (3×float32 xyz + uint32 marker)

新路线 Section 数量 7,631（旧路线 411），数据量约 18 倍。

⚠️ img_pos.txt 列含义误用

严重性：高

列3-5 是 IMU 位置数据（来自 imu.bin 列8-10），不是 LiDAR 世界坐标。如果误用于构建 transforms.json，将导致相机轨迹完全错误。

正确做法：从 calib_online.yaml 的 body_T_lidar 获取世界坐标。

此问题在旧路线项目中曾导致 V22 之前训练错误。
